WHAT THIS PROVES
  ✓ A second mutation operation type — separate from camera rename —
    works through the unifi.local.protect.* sandbox surface against
    real hardware. Specifically, the DELETE + POST pattern with array-
    style query params (?qualities=high) and array-bodied creates
    ({qualities: ['high']}).
  ✓ The HttpClient buildQueryString() correctly serialises array
    query parameters as repeated keys, matching what the Protect
    Integration API expects.
  ✓ Self-reverting mutation pattern works for non-PATCH operations
    (DELETE then POST), not just rename PATCHes. Three sequential
    ExecuteExecutor invocations (one per phase: pre-flight, mutate,
    revert) keep the QuickJS sandbox lifetimes clean.
  ✓ Pre-flight guards refuse to run unless the camera is DISCONNECTED
    AND its rtsps state is exactly "high enabled, others null". This
    is the canonical homelab Tuin state — anything else (medium/low/
    package enabled, or all null) means a human has been managing
    this camera and we should not touch it.

WHAT THIS DOES NOT PROVE
  ✗ Token-stable revert. Re-creating an RTSPS stream issues a new
    URL token; the old URL is permanently invalidated. Acceptable
    here because Tuin is DISCONNECTED so no live consumer is caching
    the old URL. The script logs "high token rotated: true" to make
    this explicit.
  ✗ Other RTSPS quality levels (medium, low, package). Only "high"
    was toggled. Multi-quality bodies are wired the same way but
    not exercised here.
  ✗ Operations against a CONNECTED camera. Out of scope for safety.

GAPS (not verified, with reasons)

  ✗ PTZ goto + return-to-home (POST /v1/cameras/{id}/ptz/goto/{slot}).
    None of the 4 cameras in the maintainer's homelab is PTZ-capable
    (probed via featureFlags — `canPtz` is absent from all four
    feature-flag dictionaries). Verification deferred until a tester
    runs this against PTZ hardware (e.g. UniFi G4 PTZ).

  ✗ Alarm-manager webhook trigger
    (POST /v1/alarm-manager/webhook/{id}). Requires an alarm to be
    pre-configured in the Protect UI with the matching ID. The
    homelab has no such alarm configured, so triggering with any ID
    is a no-op against the controller. Verification deferred until
    a tester runs this against an alarm-managed Protect deployment.

  These two gaps are tracked in the README "Verification status"
  table and CHANGELOG. The mutation script template is portable — a
  contributor with the right hardware / configuration can copy
  verify-mutations-rtsps.ts and adapt it.
