INSPECTOR VERSION CAVEAT
  We pinned to @modelcontextprotocol/inspector@0.20.0 because the
  current latest (0.21.2) has a missing-`commander` dependency error
  on Node.js v25 (see https://github.com/modelcontextprotocol/inspector
  for tracking). Either pinning to v0.20.0 OR explicitly installing
  `commander` alongside v0.21.2 works around the bug.

CACHE-SCHEMA-VERSION FINDING
  During this verification we discovered that our previously cached
  ProcessedSpec on disk (`src/spec/cache/local-v10.3.58.json` etc.)
  embeds the OUTPUT of `buildOperationIndex` — including pre-baked
  `primaryTag` strings. So a change to `normalizeTag` (like the tag-
  name compaction landed earlier today) was invisible to anyone with
  a stale cache.
  Fix included in this run: src/spec/loader.ts now writes a
  `cacheSchemaVersion` field into each cache envelope, and reads with
  a strict version check. CACHE_SCHEMA_VERSION = 2 invalidates every
  pre-compaction cache; bump on any future change to
  `buildOperationIndex` or `processSpec` shape.
  This was a real bug only the Inspector exposed: the local discover
  scripts fetch fresh on every run (forceRefresh defaulted off but
  in our flow we kept getting fresh-fetches because we'd been
  manually deleting the cache directory between iterations).
