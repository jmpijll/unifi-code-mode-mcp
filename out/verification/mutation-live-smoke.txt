WHAT THIS PROVES
  ✓ The unifi.local.protect.* sandbox surface can drive a real
    mutation operation end-to-end against Protect-enabled UniFi OS
    hardware. PATCH /v1/cameras/{id} with body { name: <new> } was
    issued through the QuickJS sandbox via path-based request(), the
    server returned the patched object, and a follow-up GET
    confirmed the persisted state.
  ✓ A round-trip rename + revert (rename to "<orig>-MCP-VERIFY-<ts>",
    GET-verify, rename back to <orig>, GET-verify) completed cleanly
    in three sequential ExecuteExecutor invocations (one per phase),
    leaving the controller in exactly its original state.
  ✓ The PATCH echo and the follow-up GET both returned the same name
    in both phases — confirming Protect's PATCH semantics are
    immediate-consistent on the camera record.
  ✓ The pre-flight safety guards behaved correctly:
      - Refuses to run unless target camera is in state=DISCONNECTED
      - Refuses to run if the original name already matches the
        MCP-VERIFY-* pattern (so a stale failed run can't self-clobber)
      - Performs the revert in a separate executor invocation even
        if the mutate-phase verify diverged, with a fatal alert
        and non-zero exit if the revert itself fails

WHAT THIS DOES NOT PROVE
  ✗ Mutation verification on the Network surface. Every Network
    create endpoint exposed in this controller's spec
    (createAclRule, createDnsPolicy, createNetwork,
    createWifiBroadcast, createTrafficMatchingList, createFirewallZone,
    createFirewallPolicy, createVouchers) requires a polymorphic
    discriminator (`$.type`, `$.management`, …) that the loaded
    OpenAPI spec does NOT expose to the synthesizer. Probing those
    discriminators blindly against live hardware is unsafe; a future
    pass of the loader will need to extract polymorphic discriminator
    enums (or we ship known-good fixture bodies per controller
    version). Network mutation verification is deferred.
  ✗ Multi-step mutation chains (create → update → delete on the same
    entity).
  ✗ End-to-end LLM-mediated mutation invocation. This run was driven
    directly from a Node script through ExecuteExecutor; an LLM
    client issuing the same PATCH via the `execute` MCP tool has not
    yet been verified.

KNOWN ARTEFACT WE COULDN'T CLEAN UP DURING DESIGN PROBING
  During the design-probing phase (before this script was finalised) we
  POSTed a test liveview to /v1/protect/integration/v1/liveviews to
  confirm that endpoint accepted creates. It did. The Protect
  Integration API does NOT expose a DELETE for liveviews — we
  exercised /v1/liveviews/{id} (404 NOT_FOUND, secondary key=DELETE),
  /v1/liveviews?id=... (same 404), and the non-Integration
  /proxy/protect/api/liveviews/{id} (401 — different auth scheme). As
  a damage-mitigation step the orphan liveview was PATCHed to
  isGlobal=false / name="_unused" / empty slots, so it is invisible
  in the default Protect dashboard. The maintainer needs to manually
  delete this liveview from the Protect web UI:

      Protect → Live View → Manage Layouts → "_unused" → Delete

  Liveview create has been removed from the candidate list for any
  future automated mutation verification; the script never creates
  liveviews.
